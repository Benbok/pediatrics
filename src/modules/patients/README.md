# Модуль Patients (Пациенты)

## 📋 Общее описание

Модуль `patients` отвечает за управление базой данных пациентов (детей) в педиатрической системе. Модуль реализует полный цикл работы с пациентами: создание, редактирование, просмотр картотеки, удаление, а также служит точкой входа для доступа к другим медицинским модулям (вакцинация, приемы).

## 🏗️ Архитектура

Модуль следует строгой многослойной архитектуре согласно `AI_CODING_GUIDELINES.md`:

```
┌─────────────────────────────────────────────────────────┐
│  Component Layer (*.tsx)                                │
│  - UI только, NO business logic                          │
│  - Обработка пользовательского ввода                    │
│  - Отображение данных                                    │
└─────────────────┬───────────────────────────────────────┘
                  │
                  ↓
┌─────────────────────────────────────────────────────────┐
│  Service Layer (src/services/patient.service.ts)       │
│  - ВСЯ бизнес-логика                                     │
│  - Zod валидация (первый уровень)                       │
│  - IPC вызовы к backend                                  │
│  - Обработка ошибок и логирование                        │
│  - Инвалидация кеша через dataEvents                    │
└─────────────────┬───────────────────────────────────────┘
                  │
                  ↓
┌─────────────────────────────────────────────────────────┐
│  IPC Handler Layer (electron/database.cjs)              │
│  - ensureAuthenticated middleware                        │
│  - Zod валидация (второй уровень - безопасность)        │
│  - Простые DB операции через Prisma                     │
│  - Шифрование персональных данных (birthDate)           │
└─────────────────┬───────────────────────────────────────┘
                  │
                  ↓
┌─────────────────────────────────────────────────────────┐
│  Database Layer (Prisma)                                │
│  - Хранение данных                                       │
│  - Миграции                                              │
│  - Связи с другими модулями (visits, vaccinations)     │
└─────────────────────────────────────────────────────────┘
```

### Принципы архитектуры

1. **Разделение ответственности**: Каждый слой имеет четко определенную роль
2. **Двойная валидация**: Frontend (UX) + Backend (Security)
3. **Централизованное логирование**: Все ошибки через `logger`
4. **Аутентификация**: Все IPC handlers защищены `ensureAuthenticated`
5. **Кеширование**: Автоматическая инвалидация кеша при изменениях через `dataEvents`
6. **Шифрование**: Персональные данные (дата рождения) шифруются на уровне базы данных

## 🎯 Основной функционал

### 1. Управление пациентами

- **Создание нового пациента**: Форма с валидацией всех полей
- **Редактирование пациента**: Изменение данных существующего пациента
- **Просмотр картотеки**: Список всех пациентов с поиском по ФИО
- **Просмотр деталей**: Детальная информация о пациенте с навигацией к модулям
- **Удаление пациента**: Удаление с подтверждением (каскадное удаление связанных данных)

### 2. Структура данных пациента

#### Основная информация
- **Фамилия** (обязательное, 2-50 символов, только кириллица)
- **Имя** (обязательное, 2-50 символов, только кириллица)
- **Отчество** (опциональное, до 50 символов, только кириллица)

#### Данные о рождении
- **Дата рождения** (обязательное, формат YYYY-MM-DD, не в будущем, возраст < 18 лет)
- **Вес при рождении** (обязательное, 500-8000 г)
- **Пол** (обязательное: `male` | `female`)

#### Системные поля
- `id` - Уникальный идентификатор
- `createdByUserId` - ID пользователя, создавшего запись
- `createdAt` - Дата создания записи

### 3. Валидация

Модуль использует **двойную валидацию**:

1. **Frontend валидация** (`patient.service.ts`):
   - Использует `ChildProfileSchema` из `validators/child.validator.ts`
   - Быстрая обратная связь для пользователя
   - Предотвращает отправку невалидных данных

2. **Backend валидация** (`database.cjs`):
   - Повторная валидация тех же схем
   - Защита от обхода frontend валидации
   - Безопасность данных

#### Правила валидации

- **Имя/Фамилия**: Минимум 2 символа, максимум 50, только кириллица, пробелы и дефисы
- **Отчество**: Максимум 50 символов, опциональное
- **Дата рождения**: 
  - Формат: YYYY-MM-DD
  - Не может быть в будущем
  - Пациент должен быть младше 18 лет
- **Вес при рождении**: От 500 до 8000 грамм
- **Пол**: Только `male` или `female`

### 4. Безопасность данных

#### Шифрование персональных данных
- **Дата рождения** (`birthDate`) шифруется на уровне базы данных
- Используется field-level encryption для защиты персональных данных (ПДн)
- Шифрование/дешифрование происходит автоматически при чтении/записи

#### Мультипользовательский доступ
- Каждый врач видит только своих пациентов (по `createdByUserId`)
- Поддержка механизма `PatientShare` для совместного доступа (в разработке)
- Разделение доступа на уровне базы данных

### 5. Интеграция с другими модулями

Модуль `patients` служит точкой входа для доступа к другим медицинским модулям:

#### Вакцинация
- Навигация: `/vaccination/{childId}`
- Управление графиком прививок
- Учет факторов риска и противопоказаний

#### Приемы (CDSS)
- Навигация: `/patients/{childId}/visits`
- История приемов
- AI-анализ симптомов
- Подбор диагнозов и терапии

### 6. Кеширование

Модуль интегрирован с системой кеширования:

- **Автоматическая инвалидация**: При создании/обновлении/удалении пациента кеш автоматически инвалидируется через `dataEvents`
- **Namespace**: `'patients'` для кеширования списка пациентов
- **Оптимизация производительности**: Снижение нагрузки на БД при частых запросах

## 🔌 Сервисы

### `patient.service.ts`

Основной сервис для работы с пациентами:

```typescript
patientService.getAllChildren()              // Получить всех пациентов
patientService.getChildById(id)              // Получить пациента по ID
patientService.createChild(data)             // Создать нового пациента
patientService.updateChild(id, updates)      // Обновить данные пациента
patientService.deleteChild(id)               // Удалить пациента
patientService.getFullName(child)            // Форматировать полное имя
patientService.getAgeLabel(birthDate)        // Получить отформатированный возраст
```

**Особенности сервиса**:
- Все методы валидируют данные через Zod
- Автоматическая инвалидация кеша через `dataEvents`
- Централизованное логирование ошибок
- Возврат понятных сообщений об ошибках

## 🗄️ База данных

### Модель Child (Prisma)

```prisma
model Child {
  id              Int      @id @default(autoincrement())
  name            String
  surname         String
  patronymic      String?
  birthDate       String   @map("birth_date")  // Зашифровано
  birthWeight     Int      @map("birth_weight")
  gender          String   // 'male' | 'female'
  createdByUserId Int?    @map("created_by_user_id")
  createdAt       DateTime @default(now()) @map("created_at")

  // Relations
  createdBy       User?              @relation(fields: [createdByUserId], references: [id])
  visits          Visit[]
  vaccinationProfile VaccinationProfile?
  vaccinationRecords VaccinationRecord[]
  allergies       PatientAllergy[]
  shares          PatientShare[]
}
```

### Связи с другими модулями

- **Visits**: Один пациент может иметь множество приемов
- **VaccinationProfile**: Один профиль вакцинации на пациента
- **VaccinationRecords**: Множество записей о вакцинации
- **PatientAllergy**: Множество аллергий пациента
- **PatientShare**: Множество записей о совместном доступе

### Каскадное удаление

При удалении пациента автоматически удаляются:
- Все приемы (`visits`)
- Профиль вакцинации (`vaccinationProfile`)
- Записи о вакцинации (`vaccinationRecords`)
- Аллергии (`allergies`)
- Записи о совместном доступе (`shares`)

## 📝 Примечания для разработчиков

1. **Всегда используйте сервисы** для бизнес-логики, не пишите её в компонентах
2. **Валидируйте данные дважды**: в сервисе и в IPC handler
3. **Логируйте ошибки** через `logger`, не используйте `console.log`
4. **Используйте dataEvents** для инвалидации кеша при изменениях
5. **Следуйте архитектуре слоев**: Component → Service → IPC → Database
6. **Учитывайте шифрование**: Дата рождения автоматически шифруется/дешифруется
7. **Проверяйте права доступа**: Убедитесь, что пользователь видит только своих пациентов

## 🔄 Миграции

При изменении структуры данных пациента:
1. Обновите `prisma/schema.prisma`
2. Создайте миграцию: `npx prisma migrate dev --name description`
3. Обновите TypeScript типы в `src/types.ts`
4. Обновите Zod схемы в `validators/child.validator.ts`
5. Обновите компоненты и сервисы
6. Убедитесь, что шифрование работает корректно
