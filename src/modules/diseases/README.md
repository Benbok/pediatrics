# Модуль Diseases (Заболевания)

## 📋 Общее описание

Модуль `diseases` отвечает за управление базой знаний заболеваний в педиатрической системе. Модуль реализует полный цикл работы с заболеваниями: создание, редактирование, загрузка клинических рекомендаций (PDF), поиск по симптомам, связь с препаратами, заметки врачей, а также предоставляет структурированные планы диагностики и лечения.

## 🏗️ Архитектура

## 🎯 Основной функционал

### 1. Управление заболеваниями

- **Создание нового заболевания**: Форма с полным набором полей
- **Редактирование заболевания**: Изменение данных существующего заболевания
- **Просмотр базы**: Список всех заболеваний с фильтрацией и поиском
- **Удаление заболевания**: Удаление с подтверждением
- **Импорт из JSON**: Импорт структурированных данных о заболевании

### 2. Структура данных заболевания

#### Основная информация
- **Название (RU)**: Обязательное поле
- **Название (EN)**: Опциональное
- **Описание**: Текстовое описание заболевания
- **ICD-10 код**: Основной код МКБ-10 (обязательное)
- **ICD-10 коды**: Массив всех связанных кодов МКБ-10

#### Клиническая информация
- **Симптомы** (`symptoms`): Массив симптомов заболевания
- **Дифференциальная диагностика** (`differentialDiagnosis`): Массив заболеваний для дифференциации
- **Красные флаги** (`redFlags`): Массив тревожных признаков

#### Структурированные планы

**План диагностики** (`diagnosticPlan`):
Массив объектов `DiagnosticPlanItem`:
- `type`: Тип исследования (`lab` | `instrumental`)
- `test`: Название исследования
- `priority`: Приоритет (`low` | `medium` | `high`)
- `rationale`: Обоснование

**План лечения** (`treatmentPlan`):
Массив объектов `TreatmentPlanItem`:
- `category`: Категория лечения (`symptomatic` | `etiologic` | `supportive` | `other`)
- `description`: Описание лечения
- `priority`: Приоритет (`low` | `medium` | `high`)

### 3. Клинические рекомендации (Guidelines)

Модуль поддерживает загрузку и работу с клиническими рекомендациями в формате PDF:

#### Загрузка рекомендаций
- **Одиночная загрузка**: Загрузка одного PDF файла
- **Пакетная загрузка**: Загрузка нескольких PDF файлов одновременно
- **Прогресс загрузки**: Отслеживание прогресса парсинга каждого файла
- **Автоматический парсинг**: Извлечение структурированных данных из PDF

#### Индексация PDF для поиска (GuidelineChunk + FTS5)

Для улучшения диагностического поиска и снижения зависимости от AI, PDF дополнительно индексируются в нормализованной таблице `GuidelineChunk`.

- **`ClinicalGuideline.chunks`**: сохраняется legacy JSON-нарезка (обратная совместимость для UI/старого поиска)
- **`GuidelineChunk`**: нормализованные чанки с метаданными
  - `guidelineId`, `diseaseId`
  - `type` (клиническая категория: `complaints`, `physicalExam`, `treatment`, ...)
  - `pageStart/pageEnd`, `sectionTitle`
  - `text`
  - `embeddingJson` (опционально)

##### FTS5: offline-first, без триггеров

Полнотекстовый поиск по `GuidelineChunk.text` делается через SQLite FTS5 виртуальную таблицу `guideline_chunks_fts`.

Важный архитектурный принцип: **FTS не синхронизируется SQLite-триггерами**. Вместо этого используется программная пересборка индекса:

- `electron/services/chunkIndexService.cjs` → `ChunkIndexService.rebuildFts()`
  - `DROP TRIGGER IF EXISTS ...` (на случай старого состояния)
  - `DROP TABLE IF EXISTS guideline_chunks_fts`
  - `CREATE VIRTUAL TABLE guideline_chunks_fts USING fts5(...)`
  - `INSERT INTO guideline_chunks_fts SELECT ... FROM guideline_chunks`

Эта стратегия устраняет классы ошибок SQLite вида `SQL logic error` при удалениях/обновлениях и гарантирует согласованность FTS с основной таблицей.

#### Структурированные секции
После парсинга PDF извлекаются следующие секции:
- `definition`: Определение заболевания
- `etiology`: Этиология
- `epidemiology`: Эпидемиология
- `classification`: Классификация
- `clinicalPicture`: Клиническая картина
- `complaints`: Жалобы
- `physicalExam`: Физикальное обследование
- `labDiagnostics`: Лабораторная диагностика
- `instrumental`: Инструментальная диагностика
- `treatment`: Лечение
- `rehabilitation`: Реабилитация
- `prevention`: Профилактика
- `medications`: Препараты (JSON массив)

#### Поиск в рекомендациях
- **Полнотекстовый поиск**: Поиск по содержимому всех загруженных PDF
- **Поиск в конкретном файле**: Фильтрация по выбранному файлу
- **Выделение результатов**: Подсветка найденных фрагментов
- **Навигация по результатам**: Переход к найденным фрагментам

#### Управление рекомендациями
- Переименование файлов
- Удаление рекомендаций
- Просмотр содержимого
- Скачивание PDF файлов

При изменении данных чанков (upload/delete/reindex) выполняется `ChunkIndexService.rebuildFts()`.

### 4. GuidelinePlan API

Модуль предоставляет нормализованный API для получения структурированных планов диагностики и лечения:

#### Источники данных (приоритет)
1. **Структурированные данные заболевания** (`disease_structured`): Данные, введенные вручную в форму
2. **Сырые данные из рекомендаций** (`guideline_raw`): Извлеченные из PDF, требуют проверки
3. **Отсутствие данных** (`none`): Планы не заполнены

#### Структура GuidelinePlan
```typescript
{
  diseaseId: number;
  diagnosticPlan: DiagnosticPlanItem[];
  treatmentPlan: TreatmentPlanItem[];
  differentialDiagnosis: string[];
  redFlags: string[];
  source: 'disease_structured' | 'guideline_raw' | 'none';
  needsReview: boolean;
  raw: {
    labDiagnostics?: string;
    instrumental?: string;
    treatment?: string;
    medications?: string;
  } | null;
}
```

#### Fallback механизм
Если структурированные планы отсутствуют, система автоматически создает планы из сырых данных рекомендаций с флагом `needsReview: true`.

### 5. Заметки врачей (Disease Notes)

Модуль поддерживает систему заметок к заболеваниям:

#### Типы заметок
- **Личные заметки**: Видны только автору
- **Общие заметки**: Видны всем пользователям (`isShared: true`)
- **Закрепленные заметки**: Отображаются вверху списка (`isPinned: true`)

#### Функционал заметок
- Создание, редактирование, удаление
- Теги для организации
- Markdown поддержка
- История изменений
- Поиск по заметкам

### 6. Связь с препаратами

- **Автоматический подбор**: Препараты подбираются по ICD-10 кодам
- **Приоритизация**: Назначение приоритета препарату для заболевания
- **Группировка**: Группировка препаратов по фармакологическим группам
- **Специфическое дозирование**: Индивидуальные правила дозирования для конкретного заболевания

### 7. Поиск по симптомам

Модуль предоставляет семантический поиск заболеваний по симптомам:

- **Входные данные**: Массив симптомов
- **Алгоритм**: Семантический поиск в базе знаний
- **Результат**: Ранжированный список заболеваний по релевантности
- **Использование**: Интеграция с AI-анализом симптомов в модуле приемов

### 8. Импорт из JSON

- Вставка JSON-структуры данных о заболевании

- Предпросмотр перед сохранением
- Предупреждения о недостающих данных

### 9. Нормализация и дедупликация диагностических тестов

Исторически одно и то же исследование могло храниться под разными именами в разных заболеваниях («Общий анализ крови», «Общий клинический анализ крови», «ОАК»). Реализована трёхуровневая защита.

#### Уровень 1 — Write-time нормализация (`electron/utils/diseaseNormalization.cjs`)

Функция `normalizeTestName(testName)` применяется ко всем тестам при каждом сохранении:
- убирает пробелы по краям
- сжимает множественные пробелы в один
- делает заглавной первую букву

Функция `_deduplicateDiagnosticPlan()` в `service.cjs` убирает точные дубликаты внутри одной болезни (ключ: `"type|test.toLowerCase()"`).

Дополнительно в `DiseaseService.upsert()` реализовано **silent auto-add** в `DiagnosticTestCatalog`:
- после успешного `create/update` болезни backend проверяет `diagnosticPlan`
- если название теста отсутствует в каталоге, создаётся новая запись (`nameRu`, `type`, `aliases: []`, `isStandard: false`)
- операция неблокирующая: при ошибке добавления в каталог болезнь всё равно сохраняется, ошибка только логируется
- после `upsert` IPC handler инвалидирует кэш `diagnostic_test_catalog`, поэтому при следующей загрузке формы новые названия доступны в автодополнении

#### Уровень 2 — Fuzzy-matching при импорте JSON

`normalizeDiseaseData(data, availableTestNames?)` принимает необязательный список всех известных названий тестов. Для каждого входящего теста вычисляется схожесть по **Jaro-Winkler**:

```
jaroWinkler(inputName, candidate) >= 0.92 → заменяется каноническим именем
```

При `upsert()` backend автоматически подгружает все существующие названия тестов из БД и передаёт их в `normalizeDiseaseData`, поэтому импорт JSON автоматически выравнивается по уже сохранённым именам.

#### Уровень 3 — UI автодополнение (`TestNameAutocomplete.tsx`)

В форме редактирования болезни поле `test` в плане диагностики заменено на `<TestNameAutocomplete>`:
- при монтировании формы загружается список канонических названий из `DiagnosticTestCatalog` через `diseaseService.getDiagnosticCatalogTestNames()`
- при вводе фильтрует список по подстроке (без учёта регистра), показывает до 12 подсказок
- выбор из дропдауна вставляет каноническое имя
- при `blur` выполняется проверка по алиасам через `diseaseService.resolveDiagnosticTestName()`; если найдено соответствие, поле автоматически заменяется на каноническое имя
- если совпадений нет — разрешается ввести новое имя; после сохранения болезни это имя автоматически добавляется в каталог (silent auto-add)

#### Модель `DiagnosticTestCatalog` (Prisma)

Добавлена в Phase 3 как постоянный справочник:

```prisma
model DiagnosticTestCatalog {
  id         Int      @id @default(autoincrement())
  nameRu     String   @unique @map("name_ru")   // каноническое имя
  type       String                              // 'lab' | 'instrumental'
  aliases    String   @default("[]")             // JSON: список вариантов написания
  isStandard Boolean  @default(false)            // из официальных стандартов?
  createdAt  DateTime @default(now())
  updatedAt  DateTime @updatedAt

  @@index([type])
  @@map("diagnostic_test_catalog")
}
```

Миграция: `prisma/migrations/20260331120000_add_diagnostic_test_catalog/migration.sql`

#### Пополнение каталога (алиасы)

Все записи каталога хранятся в массиве `CATALOG` в файле [scripts/seed-diagnostic-catalog.py](../../../scripts/seed-diagnostic-catalog.py). Скрипт идемпотентен: повторный запуск обновляет существующие записи и добавляет новые, ничего не удаляя.

**Добавить новый тест:**

```python
{
    "name_ru": "Анализ кала на скрытую кровь",  # каноническое имя (как будет сохранено в БД)
    "type": "lab",                               # 'lab' или 'instrumental'
    "aliases": '["Скрытая кровь в кале", "Проба Грегерсена", "Реакция Грегерсена"]',
    "is_standard": 1,
},
```

**Добавить алиас к существующему тесту** — найти нужную запись и расширить строку `aliases`:

```python
# Было:
"aliases": '["ОАК", "Анализ крови"]',
# Стало:
"aliases": '["ОАК", "Анализ крови", "Гемограмма развёрнутая"]',
```

После любых правок применить:

```powershell
python scripts/seed-diagnostic-catalog.py
```

Вывод покажет количество созданных и обновлённых записей (`Created: N, Updated: M`).

> **Важно**: Node.js версия (`seed-diagnostic-catalog.cjs`) не работает из-за несовместимости `better-sqlite3` с системным Node. Используйте только `.py` версию.

### 10. Валидация

Модуль использует **двойную валидацию**:

1. **Frontend валидация** (`diseaseService.ts`):
   - Использует `DiseaseSchema` из `validators/disease.validator.ts`
   - Быстрая обратная связь для пользователя
   - Предотвращает отправку невалидных данных

2. **Backend валидация** (`handlers.cjs`):
   - Повторная валидация тех же схем
   - Защита от обхода frontend валидации
   - Безопасность данных

#### Схемы валидации

- `DiseaseSchema`: Основная схема заболевания
- `DiagnosticPlanItemSchema`: Схема элемента плана диагностики
- `TreatmentPlanItemSchema`: Схема элемента плана лечения

Основной сервис для работы с заболеваниями:

```typescript
diseaseService.getDiseases()                    // Получить все заболевания

diseaseService.uploadGuideline(diseaseId, pdfPath) // Загрузить рекомендацию
diseaseService.uploadGuidelinesBatch(...)     // Пакетная загрузка рекомендаций
diseaseService.uploadGuidelinesAsync(...)      // Асинхронная очередь загрузки
diseaseService.updateGuideline(id, title)     // Обновить рекомендацию

diseaseService.importFromJson(jsonString)     // Импорт из JSON
```

## 📝 Примечания для разработчиков

1. **Всегда используйте сервисы** для бизнес-логики, не пишите её в компонентах
2. **Валидируйте данные дважды**: в сервисе и в IPC handler
3. **Логируйте ошибки** через `logger`, не используйте `console.log`
4. **Используйте dataEvents** для инвалидации кеша при изменениях
5. **Следуйте архитектуре слоев**: Component → Service → IPC → Database
6. **Парсите JSON поля** только на границе с базой данных (в backend service)
7. **Используйте GuidelinePlan API**: Для получения нормализованных планов диагностики и лечения
8. **Нормализуйте данные**: Используйте `normalizeDisease` перед использованием данных
9. **Никогда не сохраняйте название теста вручную** без прохода через `normalizeTestName()` — используйте `normalizeDiseaseData()` или компонент `TestNameAutocomplete`
10. **Не дублируйте дедупликацию**: логика дедупликации живёт в `diseaseNormalization.cjs` и `service.cjs`, не копируйте её в компоненты

## 🔄 Миграции

При изменении структуры данных заболевания:
1. Обновите `prisma/schema.prisma`
2. Создайте миграцию: `npx prisma migrate dev --name description`  
   > ⚠️ Если в БД есть FTS5-таблицы (`guideline_chunks_fts`), Prisma обнаружит drift и потребует сброс. В этом случае используйте ручной подход (см. `MIGRATION_INSTRUCTIONS.md`): создайте `migration.sql` вручную, примените через Python/SQLite, затем `npx prisma migrate resolve --applied <name>`
3. Обновите TypeScript типы в `src/types.ts`
4. Обновите Zod схемы в `validators/disease.validator.ts`
5. Обновите компоненты и сервисы
6. Обеспечьте обратную совместимость при изменении JSON полей
7. Обновите шаблон импорта из JSON при необходимости
