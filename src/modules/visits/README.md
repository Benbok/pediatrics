# Модуль Visits (Приемы)

## 📋 Общее описание

Модуль `visits` отвечает за управление медицинскими приемами пациентов в педиатрической системе. Модуль реализует полный цикл работы с приемами: создание, редактирование, просмотр истории, AI-анализ симптомов, подбор диагнозов и назначение терапии.

## 🏗️ Архитектура

Модуль следует строгой многослойной архитектуре согласно `AI_CODING_GUIDELINES.md`:

```

│  Component Layer (*.tsx)                                │
│  - UI только, NO business logic                          │

└─────────────────┬───────────────────────────────────────┘
                  │
                  ↓
┌─────────────────────────────────────────────────────────┐
│  Service Layer (*.service.ts)                           │
│  - ВСЯ бизнес-логика                                     │
│  - Zod валидация (первый уровень)                       │
│  - IPC вызовы к backend                                  │
│  - Обработка ошибок и логирование                        │
└─────────────────┬───────────────────────────────────────┘
                  │
                  ↓
┌─────────────────────────────────────────────────────────┐
│  IPC Handler Layer (electron/modules/visits/*.cjs)      │
│  - ensureAuthenticated middleware                        │
│  - Zod валидация (второй уровень - безопасность)        │
│  - Простые DB операции через Prisma                     │
└─────────────────┬───────────────────────────────────────┘
                  │
                  ↓
┌─────────────────────────────────────────────────────────┐

│  - Хранение данных                                       │

└─────────────────────────────────────────────────────────┘
```

### Принципы архитектуры

1. **Разделение ответственности**: Каждый слой имеет четко определенную роль
2. **Двойная валидация**: Frontend (UX) + Backend (Security)
3. **Централизованное логирование**: Все ошибки через `logger`
4. **Аутентификация**: Все IPC handlers защищены `ensureAuthenticated`

## 🎯 Основной функционал

### 🔬 CDSS - Система поддержки принятия клинических решений

Модуль интегрирует продвинутую **CDSS систему** для анализа жалоб и автоматического подбора диагнозов, препаратов и исследований.

⭐ **Подробная документация**: Смотрите [CDSS_SYSTEM.md](./CDSS_SYSTEM.md) для полного описания функционала

**Основные возможности CDSS:**
- 🤖 **AI-анализ жалоб** через Gemini API с парсингом симптомов
- 🏥 **Дифференциальный диагноз** с семантическим поиском и ранжированием
- 💊 **Автоматический подбор препаратов** с расчетом дозировок по весу/возрасту
- 🧪 **Рекомендации диагностических исследований** по МКБ кодам
- ⚠️ **Учет аллергий и противопоказаний** при подборе терапии
- 📊 **Контекстное ранжирование** с учетом возраста, витальных показателей

### 1. Управление приемами

- **Создание нового приема**: Форма с полным набором полей (025/у) + дополнительные разделы (план обследования, рекомендации)
- **Редактирование приема**: Изменение данных существующего приема
- **Просмотр истории**: Список всех приемов пациента с фильтрацией и поиском
- **Удаление приема**: Удаление с подтверждением

### 2. Типы приемов

Модуль поддерживает следующие типы приемов:

- **primary** - Первичный прием
- **followup** - Повторный прием
- **consultation** - Консультация
- **emergency** - Неотложный прием
- **urgent** - Срочный прием

Тип приема влияет на:
- Отображение разделов формы (например, анамнез жизни только для первичного приема и консультации)
- Логику работы с шаблонами
- Печатные формы

В backend дополнительно реализована логика определения типа приёма на основе истории (первичный/повторный) и наличия направления (консультация).

### 3. Структура данных приема

Прием включает следующие разделы:

#### Организационные данные
- Дата/время приема (`visitDate`, `visitTime`)
- Тип приема (`visitType`)
- Место приема (`visitPlace`: clinic/home/other)
- Номер талона (`ticketNumber`) и следующая дата визита (`nextVisitDate`) при наличии

#### Антропометрия
- Вес (кг) с автоматическим расчетом ИМТ
- Рост (см) с автоматическим расчетом ППТ (площадь поверхности тела)
- Категория ИМТ по возрасту

#### Анамнез жизни (форма 025/у)
**Отображается только для первичного приема и консультации**

- **Наследственность**: Туберкулез, диабет, гипертония, онкология, аллергии, прочее
- **Сведения о беременности и родах**:
  - Течение беременности
  - Акушерский анамнез
  - Способ родоразрешения (естественные/кесарево)
  - Длительность беременности (недель, формат: недели.дни, например 38.5)
  - Масса и рост при рождении
  - Оценка по шкале Апгар
  - Осложнения периода новорожденности
- **Вскармливание**:
  - Тип: Грудное / Искусственное / Смешанное
  - Длительность грудного вскармливания
  - Название молочной смеси (для искусственного/смешанного)
  - Возраст введения прикормов
  - Особенности питания
- **Перенесенные инфекционные заболевания**: Корь, ветрянка, краснуха, коклюш, скарлатина, ангина (с частотой), прочие
- **Аллергический статус**: Пищевая, лекарственная, на материалы, на укусы, сезонная аллергии

**Хранение**: Данные анамнеза жизни хранятся в JSON полях базы данных для структурированности и удобства обработки.

#### Анамнез заболевания
- Жалобы на момент поступления
- Когда началось заболевание и первые симптомы
- Течение болезни
- Лечение, проводимое до обращения

#### Показатели жизнедеятельности (Vital Signs)
- Артериальное давление (систолическое/диастолическое) с возрастными нормами
- Пульс (ЧСС) с возрастными нормами
- Частота дыхательных движений (ЧДД)
- Температура тела
- Сатурация кислорода (SpO2)
- Уровень сознания

#### Объективный осмотр
- Структурированный осмотр по системам (JSON)
- Текстовое описание общего состояния
- Шаблоны текста осмотра

#### Диагнозы
- **Основной диагноз**: Обязательное поле, связь с базой знаний заболеваний
- **Осложнения**: Множественный выбор
- **Сопутствующие диагнозы**: Множественный выбор

#### План обследования
- Лабораторные исследования (`laboratoryTests`)
- Инструментальные исследования (`instrumentalTests`)
- Консультации специалистов (`consultationRequests`)
- Физиотерапия (`physiotherapy`)

#### Назначения (Prescriptions)
- Список препаратов с дозировками
- Расчет дозировок на основе веса/возраста пациента
- Шаблоны назначений
- Маршрут введения, разведение, кратность приема

#### Рекомендации
- Список рекомендаций (`recommendations`) с поддержкой шаблонов

### 4. AI-интеграция (CDSS)

#### Анализ симптомов
- Кнопка "Анализ AI" в разделе "Анамнез заболевания"
- Анализ жалоб и анамнеза с помощью Google Gemini API
- Парсинг симптомов из текста
- Семантический поиск в базе знаний заболеваний
- Ранжирование диагнозов по релевантности

#### Диагностический поиск по клиническим рекомендациям (PDF)

CDSS использует не только симптомы заболевания, но и текст из многостраничных PDF клинических рекомендаций (модуль `diseases`).

Основные источники кандидатов:

1. **Симптомный поиск** (семантический, если доступен Gemini embeddings)
2. **Поиск по клиническим чанкам `GuidelineChunk`**
   - лексический: SQLite FTS5 (`guideline_chunks_fts`)
   - семантический (опционально): in-memory `ChunkIndexService` по `embeddingJson`

#### Новый пайплайн `analyzeVisit()` (backend)

Реализация: `electron/modules/visits/service.cjs` → `analyzeVisit()`.

Пайплайн:

1. **Сбор клинического контекста**
   - жалобы + анамнез заболевания
   - анамнез жизни (через `AnamnesisFormatter`)
   - витальные показатели и отклонения

2. **Извлечение симптомов**
   - `cdssService.parseComplaints()`

3. **Clinical Query Builder**
   - `electron/services/clinicalQueryBuilder.cjs` → `buildClinicalQuery(visit)`
   - формирует компактный запрос, включая аномальные витальные показатели

4. **Candidate Search (dual-path)**
   - `electron/services/cdssSearchService.cjs` → `CDSSSearchService.searchByClinicalData(clinicalQuery, symptoms)`
   - включает structured pre-filter, FTS5 поиск по чанкам и опциональную семантику

5. **Ranking**
   - `electron/services/cdssRankingService.cjs` → `rankDiagnosesWithContext(symptoms, candidates, patientContext, clinicalQuery)`
   - AI-ранжирование опционально, с graceful fallback

#### Offline-first и деградация

- При недоступности Gemini (квоты/geo/сеть) система продолжает работать за счёт:
  - FTS5 (`GuidelineChunk.text`)
  - keyword/symptom fallback
- Embeddings являются опциональными: при их отсутствии semantic path пропускается.

#### Форматирование данных для AI
- `AnamnesisFormatter` преобразует структурированный анамнез жизни (JSON) в читаемый текст
- Включает все разделы: наследственность, перинатальный анамнез, вскармливание, инфекции, аллергии
- Учитывает клинически значимые данные (недоношенность, ранний прикорм, частые ангины и т.д.)

#### Рекомендации по терапии
- Автоматический подбор препаратов для выбранного диагноза
- Расчет дозировок на основе веса и возраста пациента
- Отображение рекомендаций с возможностью добавления в назначения

### 5. Шаблоны

#### Шаблоны приемов
- Создание шаблонов для быстрого заполнения повторяющихся данных
- Применение шаблона к новому приему
- Редактирование и удаление шаблонов
- При создании шаблона можно перезаписать существующий

#### Шаблоны назначений
- Создание наборов препаратов для частых диагнозов
- Пакетное добавление препаратов из шаблона
- Пакетное редактирование дозировок в шаблоне
- Создание шаблона из текущих назначений

#### Шаблоны плана обследования
- Шаблоны лабораторных/инструментальных исследований и консультаций
- Пакетное редактирование элементов плана

#### Шаблоны рекомендаций
- Шаблоны списков рекомендаций (личные + публичные)
- Быстрое применение к текущему приёму

#### Шаблоны текста осмотра
- Быстрое добавление стандартных формулировок объективного осмотра

### 6. Валидация

Модуль использует **двойную валидацию**:

1. **Frontend валидация** (`visitService.ts`):
   - Использует `VisitSchema` из `validators/visit.validator.ts`
   - Быстрая обратная связь для пользователя
   - Предотвращает отправку невалидных данных

2. **Backend валидация** (`handlers.cjs`):
   - Повторная валидация тех же схем
   - Защита от обхода frontend валидации
   - Безопасность данных

#### Схемы валидации

- `VisitSchema`: Основная схема приема
- `AnalyzeVisitSchema`: Схема для AI-анализа
- `HeredityDataSchema`, `BirthDataSchema`, `FeedingDataSchema`, и др.: Схемы для структурированных данных анамнеза

### 7. Печатные формы

- **Форма 025/у-1**: Печатная форма приема согласно приказу Минздрава
- Компонент `VisitTicket025_1.tsx` для генерации печатной формы

## 🔌 Сервисы

### `visitService.ts`

Основной сервис для работы с приемами:

```typescript
visitService.getVisits(childId)              // Получить все приемы пациента
visitService.getVisit(id)                     // Получить прием по ID
visitService.upsertVisit(data)                // Создать/обновить прием
visitService.deleteVisit(id)                  // Удалить прием
visitService.analyzeVisit(visitId)           // AI-анализ симптомов
visitService.getMedicationsForDiagnosis(diseaseId)  // Препараты для диагноза
visitService.getMedicationsByIcdCode(icdCode)    // Препараты по коду МКБ
```

## 📝 Примечания для разработчиков

1. **Всегда используйте сервисы** для бизнес-логики, не пишите её в компонентах
2. **Валидируйте данные дважды**: в сервисе и в IPC handler
3. **Логируйте ошибки** через `logger`, не используйте `console.log`
4. **Используйте транзакции** для связанных операций
5. **Парсите JSON поля** только на границе с базой данных (в backend service)
6. **Следуйте архитектуре слоев**: Component → Service → IPC → Database

## 🔄 Миграции

При изменении структуры данных приема:
1. Обновите `prisma/schema.prisma`
2. Создайте миграцию: `npx prisma migrate dev --name description`
3. Обновите TypeScript типы в `src/types.ts`
4. Обновите Zod схемы в `validators/`
5. Обновите компоненты и сервисы
6. Обеспечьте обратную совместимость при изменении JSON полей
