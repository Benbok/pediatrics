# Модуль Medications (Препараты)

## 📋 Общее описание

Модуль `medications` отвечает за управление базой данных лекарственных препаратов в педиатрической системе. Модуль реализует полный цикл работы с препаратами: создание, редактирование, импорт из Видаль через AI, расчет детских дозировок, связь с заболеваниями, фильтрацию и поиск.

## 🏗️ Архитектура

Модуль следует строгой многослойной архитектуре согласно `AI_CODING_GUIDELINES.md`:

### Принципы архитектуры

1. **Разделение ответственности**: Каждый слой имеет четко определенную роль
2. **Двойная валидация**: Frontend (UX) + Backend (Security)
3. **Централизованное логирование**: Все ошибки через `logger`
4. **Аутентификация**: Все IPC handlers защищены `ensureAuthenticated`
5. **Кеширование**: Автоматическая инвалидация кеша при изменениях через `dataEvents`
6. **AI-интеграция**: Парсинг данных из Видаль через Google Gemini API

## 🎯 Основной функционал

### 2. Импорт из Видаль

Модуль поддерживает автоматический импорт данных о препаратах из справочника Видаль:

#### Импорт по URL
- Ввод URL страницы препарата на сайте Видаль
- Автоматическое извлечение HTML-контента
- AI-парсинг через Google Gemini API
- Валидация и проверка данных
- Предпросмотр перед сохранением

#### Импорт из JSON
- Вставка JSON-структуры данных
- Валидация структуры
- Проверка на дубликаты
- Предпросмотр перед сохранением

#### Массовый импорт из локальной vidal.db
- Источник: локальная база `C:/Users/Arty/Desktop/ru.medsolutions/vidal.db`
- Скрипт: `python scripts/import-vidal-medications.py`
- Уровень импорта: один `Document` = одна запись `Medication`
- Повторный запуск безопасен: скрипт идемпотентный
- При повторном запуске обновляются Vidal-поля и справочные поля, но не затираются вручную заполненные `pediatricDosing`, `adultDosing`, `forms`, `icd10Codes`
- Если у `Document` нет связанной `MoleculeName`, используется fallback: `Document.RusName` → `activeSubstance`

#### Первая волна бронхолитиков из Vidal
- Генератор: `python scripts/generate_bronchodilator_jsons.py`
- Целевая директория JSON: `src/modules/medications/data/bronchodilators/`
- Объём первой волны: `49` агрегатов `name_ru + atc_code`
- Стратегия импорта: `python scripts/import_medications_json_batch.py --dir src/modules/medications/data/bronchodilators --db prisma/dev.db --upsert-by name_ru_atc --strict-keys`
- Для ингаляционных растворов важна корректная нормализация формы как `solution`, иначе структурированные `pediatricDosing` не применяются к препаратам уровня `Беродуал ®` и `Атровент ®`
- Первая волна сознательно исключает контроллерные комбинации `R03AK*`; импортированы только строгие бронходилататорные группы `R03AC*`, `R03AL*`, `R03BB*`, `R03CC*`, `R03DA*`, `R03DB*`

#### Валидация импорта
- Проверка обязательных полей
- Валидация структуры дозировок
- Проверка корректности форм выпуска
- Предупреждения о недостающих данных
- Индикация необходимости ручной проверки

### 3. Структура данных препарата

#### Основная информация
- **Название (RU)**: Обязательное поле
- **Название (EN)**: Опциональное
- **Действующее вещество**: Обязательное
- **ATC-код**: Код анатомо-терапевтической классификации
- **Производитель**: Название производителя

#### Формы выпуска
Массив объектов `MedicationForm`:
- `type`: Тип формы (tablet, solution, syrup, injection, и т.д.)
- `concentration`: Концентрация
- `strengthMg`: Сила в мг
- `mgPerMl`: Мг на мл (для растворов)
- `volumeMl`: Объем в мл
- `description`: Описание формы

#### Дозирование
- **Педиатрическое дозирование** (`pediatricDosing`): Массив правил дозирования для детей
  - `weight_based`: На основе веса (мг/кг)
  - `bsa_based`: На основе площади поверхности тела (мг/м²)
  - `age_based`: На основе возраста
  - `fixed`: Фиксированная доза
- **Взрослое дозирование** (`adultDosing`): Опционально

#### Ограничения дозирования
- `minInterval`: Минимальный интервал между дозами (часы)
- `maxDosesPerDay`: Максимальное количество доз в сутки
- `maxDurationDays`: Максимальная длительность приема (дни)

#### Клиническая информация
- **Показания** (`indications`): Массив показаний к применению
- **Противопоказания** (`contraindications`): Текстовое описание
- **Осторожность** (`cautionConditions`): Условия, требующие осторожности
- **Побочные эффекты** (`sideEffects`): Описание побочных эффектов
- **Взаимодействия** (`interactions`): Взаимодействие с другими препаратами
- **Беременность** (`pregnancy`): Применение при беременности
- **Лактация** (`lactation`): Применение при лактации

#### Дополнительные поля Vidal
- **OTC-флаг** (`isOtc`): безрецептурный препарат
- **Передозировка** (`overdose`): текст из Vidal
- **Применение у детей** (`childDosing`, `childUsing`): текст + статус `Can | Care | Not | Qwes`
- **Почечная недостаточность** (`renalInsuf`, `renalUsing`): текст + статус
- **Печеночная недостаточность** (`hepatoInsuf`, `hepatoUsing`): текст + статус
- **Особые указания** (`specialInstruction`): HTML-текст из Vidal
- **Фармакокинетика** (`pharmacokinetics`): HTML-текст из Vidal
- **Фармакодинамика** (`pharmacodynamics`): HTML-текст из Vidal

#### Классификация
- **ICD-10 коды** (`icd10Codes`): Массив кодов МКБ-10 для связи с заболеваниями
- **Клинико-фармакологическая группа** (`clinicalPharmGroup`): Основная группа
- **Фармакотерапевтическая группа** (`pharmTherapyGroup`): Подгруппа

#### Маршрут введения
- `routeOfAdmin`: oral, rectal, iv, im, sublingual, topical, inhalation, iv_bolus, iv_infusion, iv_slow, sc, intranasal, transdermal

#### Организация
- **Избранное** (`isFavorite`): Флаг избранного препарата
- **Теги** (`userTags`): Массив пользовательских тегов
- **Счетчик использования** (`usageCount`): Количество использований
- **Последнее использование** (`lastUsedAt`): Дата последнего использования

### 4. Расчет дозировок

Модуль предоставляет автоматический расчет детских дозировок:

#### Параметры расчета
- Вес пациента (кг)
- Возраст (месяцы)
- Рост (см, опционально) - для расчета BSA

#### Типы расчетов
- **По весу**: `мг/кг × вес = доза`
- **По площади тела (BSA)**: `мг/м² × BSA = доза`
- **По возрасту**: Фиксированная доза для возрастной группы
- **Фиксированная**: Постоянная доза независимо от параметров

#### Результат расчета
- Рекомендуемая доза (мг)
- Кратность приема
- Максимальная суточная доза
- Предупреждения о превышении доз
- Информация о формах выпуска

### 5. Калькулятор разведения

Для инфузионных форм препаратов:

- **Концентрация ампулы**: Мг/мл
- **Объем разведения**: Мл растворителя
- **Длительность инфузии**: Минуты
- **Результат**:
  - Объем препарата для введения (мл)
  - Общий объем инфузии (мл)
  - Скорость инфузии (мл/ч)

### 6. Связь с заболеваниями

- **Связь по ICD-10**: Автоматический подбор препаратов для диагноза
- **Правило сопоставления ICD-10**:
  - Препарат показывается, если есть **хотя бы одно** совпадение между `medication.icd10Codes[]` и кодами заболевания
  - Поддерживается **точное совпадение**: `J10.8` ↔ `J10.8`
  - Поддерживается **префиксное совпадение (общий код препарата → уточненный код заболевания)**: `J10` ↔ `J10.8`
  - Пример: если у препарата есть `J10`, а у заболевания есть `J10.8`, препарат считается подходящим и попадает в рекомендации
- **Приоритет**: Назначение приоритета препарату для заболевания
- **Специфическое дозирование**: Индивидуальные правила дозирования для конкретного заболевания
- **Длительность**: Рекомендуемая длительность приема

### 7. Фильтрация и поиск

#### Поиск
- По названию (RU/EN)
- По действующему веществу
- По ATC-коду
- По производителю

#### Фильтры
- **Фармакологическая группа**: Фильтр по клинико-фармакологической группе
- **Форма выпуска**: Фильтр по типу формы (таблетки, раствор, и т.д.)
- **Избранное**: Показать только избранные препараты
- **Теги**: Фильтрация по пользовательским тегам

### 8. Валидация

Модуль использует **двойную валидацию**:

1. **Frontend валидация** (`medicationService.ts`):
   - Использует `MedicationSchema` из `validators/medication.validator.ts`
   - Быстрая обратная связь для пользователя
   - Предотвращает отправку невалидных данных

2. **Backend валидация** (`handlers.cjs`):
   - Повторная валидация тех же схем
   - Защита от обхода frontend валидации
   - Безопасность данных

#### Схемы валидации

- `MedicationSchema`: Основная схема препарата
- `CalculateDoseSchema`: Схема для расчета дозировки
- `LinkMedicationToDiseaseSchema`: Схема для связи с заболеванием

### 9. История изменений

Полный audit trail всех изменений препарата:

- Дата и время изменения
- Пользователь, внесший изменение
- Тип изменения (создание, обновление, удаление)
- Измененные поля
- Старые и новые значения
- Источник изменения (manual, vidal_import)

## 🔌 Сервисы

### `medicationService.ts`

Основной сервис для работы с препаратами:

```typescript
medicationService.getMedications()                    // Получить все препараты
medicationService.getMedication(id)                  // Получить препарат по ID
medicationService.upsertMedication(data, source)     // Создать/обновить препарат
medicationService.deleteMedication(id)               // Удалить препарат
medicationService.calculateDose(...)                  // Рассчитать дозировку
medicationService.linkToDisease(...)                  // Связать с заболеванием
medicationService.getMedicationsByDisease(diseaseId) // Получить препараты для заболевания
medicationService.checkDuplicate(name, excludeId)    // Проверить дубликаты
medicationService.importFromVidal(url)              // Импорт из Видаль (URL)
medicationService.importFromJson(jsonString)         // Импорт из JSON
medicationService.getPharmacologicalGroups()         // Получить все группы
medicationService.getFormTypes()                     // Получить все типы форм
medicationService.searchByGroup(groupName)          // Поиск по группе
medicationService.toggleFavorite(id)                 // Переключить избранное
medicationService.addTag(id, tag)                   // Добавить тег
medicationService.getChangeHistory(id)              // Получить историю изменений
```

**Особенности сервиса**:
- Все методы валидируют данные через Zod
- Автоматическая инвалидация кеша через `dataEvents`
- Централизованное логирование ошибок
- Нормализация маршрутов введения
- Возврат понятных сообщений об ошибках

## 📝 Примечания для разработчиков

1. **Всегда используйте сервисы** для бизнес-логики, не пишите её в компонентах
2. **Валидируйте данные дважды**: в сервисе и в IPC handler
3. **Логируйте ошибки** через `logger`, не используйте `console.log`
4. **Используйте dataEvents** для инвалидации кеша при изменениях
5. **Следуйте архитектуре слоев**: Component → Service → IPC → Database
6. **Парсите JSON поля** только на границе с базой данных (в backend service)
7. **Валидируйте AI-импорт**: Всегда проверяйте данные, импортированные через AI
8. **Нормализуйте маршруты**: Используйте `normalizeMedicationRoutes` перед сохранением

## 🔄 Миграции

При изменении структуры данных препарата:
1. Обновите `prisma/schema.prisma`
2. Для обычных изменений создайте миграцию. Для текущего проекта учитывайте drift из-за FTS-таблиц: используйте ручной SQL + apply script + `prisma migrate resolve --applied`, если `migrate dev` блокируется
3. Обновите TypeScript типы в `src/types.ts`
4. Обновите Zod схемы в `validators/medication.validator.ts`
5. Обновите компоненты и сервисы
6. Обеспечьте обратную совместимость при изменении JSON полей
7. Обновите шаблон импорта из Видаль при необходимости
