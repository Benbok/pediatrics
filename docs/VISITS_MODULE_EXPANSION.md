# Расширение модуля Приемы согласно формам МЗ РФ

## Обзор

Модуль "Приемы" был полностью переработан для соответствия требованиям Приказа Минздрава России от 13 мая 2025 г. № 274н. Реализованы унифицированные формы первичной медицинской документации (формы 025/у и 025-1/у) с полной поддержкой структурированного анамнеза, показателей жизнедеятельности, объективного осмотра по системам органов и информированного согласия.

## Новые возможности

### 1. Типы приемов

Поддержка различных типов приемов с автоматическим определением:
- **Первичный** (primary) - первый прием по данному заболеванию
- **Повторный** (followup) - повторный прием для наблюдения
- **Консультация** (consultation) - консультация специалиста
- **Экстренный** (emergency) - экстренная медицинская помощь
- **Неотложный** (urgent) - неотложная медицинская помощь

### 2. Структурированный анамнез

Разделы анамнеза:
- Анамнез заболевания (начало, развитие симптомов, предшествующее лечение)
- Анамнез жизни (семейный анамнез, особенности беременности и родов)
- Аллергологический анамнез (уточненные аллергические реакции)
- Перенесенные заболевания

### 3. Показатели жизнедеятельности

Автоматический расчет нормативов по возрасту ребенка:
- Артериальное давление (систолическое/диастолическое)
- Пульс с нормами по возрасту
- Температура тела с индикацией лихорадки
- Частота дыхательных движений (ЧДД) с нормами
- Сатурация кислорода (SpO2)
- Уровень сознания

### 4. Объективный осмотр по системам

Структурированный осмотр с аккордеон-секциями:
- Общее состояние
- Сознание
- Кожные покровы и видимые слизистые
- Лимфатические узлы
- Костно-мышечная система
- Органы дыхания
- Сердечно-сосудистая система
- Органы брюшной полости
- Мочеполовая система
- Нервная система

### 5. Диагнозы с интеграцией МКБ

Три способа ввода диагнозов:
1. **Ручной ввод** - ввод кода МКБ и названия вручную
2. **Выбор из справочника МКБ** - поиск и выбор из полного справочника МКБ-10
3. **Выбор из базы знаний** - выбор из заболеваний в базе знаний с автоматическим заполнением кода и названия

Поддержка:
- Основной диагноз
- Осложнения (множественный выбор)
- Сопутствующие заболевания (множественный выбор)

### 6. Информированное согласие

Полная поддержка информированного добровольного согласия на медицинское вмешательство (Приказ Минздрава России № 1051н):
- Описание вмешательства
- Цели и ожидаемые результаты
- Альтернативные методы лечения
- Риски и осложнения
- Статус согласия (дано/отказано/отозвано)
- Подписи пациента/родителя и врача
- Шаблоны для разных типов вмешательств

### 7. Система шаблонов

Создание и применение шаблонов приемов:
- Шаблоны по типам приемов
- Публичные и приватные шаблоны
- Шаблоны по умолчанию
- Автозаполнение при применении шаблона

### 8. AI-интеграция

Расширенный AI-анализ с учетом:
- Жалоб
- Анамнеза заболевания
- Аллергологического анамнеза
- Показателей жизнедеятельности
- Объективного осмотра по системам

Результаты:
- Список вероятных заболеваний с обоснованием
- Рекомендованные препараты с учетом аллергоанамнеза
- План лечения

### 9. Workflow с просмотром карточек заболеваний

Возможность открыть карточку заболевания из списка AI-рекомендаций в боковой панели без потери введенных данных в форме приема.

### 10. Формы 025/у и 025-1/у

- **Форма 025/у** - Медицинская карта пациента (основная форма приема)
- **Форма 025-1/у** - Талон пациента (для каждого обращения)

Поддержка печати и экспорта в PDF.

## Техническая архитектура

### База данных

**Новые модели:**
- `InformedConsent` - информированное согласие
- `VisitTemplate` - шаблоны приемов

**Расширенная модель Visit:**
- Типы приемов и организационные данные
- Структурированный анамнез (4 поля)
- Показатели жизнедеятельности (7 полей)
- Объективный осмотр по системам (10 полей)
- Диагнозы (новая структура с поддержкой МКБ)
- Исходы и маршрутизация
- Документооборот

### Валидация

Двойная валидация:
1. **Frontend** (`src/validators/visit.validator.ts`) - для UX
2. **Backend** (`electron/modules/visits/service.cjs`) - для безопасности

Правила валидации по типам приемов:
- Первичный: жалобы, дата, диагноз (при завершении)
- Повторный: жалобы, дата
- Консультация: жалобы, направивший врач, диагноз (при завершении)
- Экстренный/Неотложный: жалобы или диагноз, время, показатели жизнедеятельности

### Сервисы

**Backend:**
- `VisitService` - расширен с логикой типов приемов, нормализацией диагнозов, генерацией талонов
- `InformedConsentService` - управление согласиями
- `VisitTemplateService` - управление шаблонами

**Frontend:**
- `visitService` - обновлен для новых полей
- `informedConsentService` - новый сервис для согласий

### Компоненты UI

**Основные компоненты:**
- `VisitTypeSelector` - выбор типа приема
- `AnamnesisSection` - структурированный анамнез
- `VitalSignsSection` - показатели жизнедеятельности
- `PhysicalExamBySystems` - объективный осмотр по системам
- `DiagnosisSelector` - выбор диагноза (основной)
- `MultipleDiagnosisSelector` - множественный выбор (осложнения, сопутствующие)
- `IcdCodeSearchModal` - поиск в справочнике МКБ
- `DiseaseSearchModal` - поиск в базе знаний заболеваний
- `DiseaseSidePanel` - просмотр карточки заболевания
- `InformedConsentForm` - форма информированного согласия
- `VisitTicket025_1` - талон пациента (форма 025-1/у)
- `VisitTemplateSelector` - выбор шаблона приема

## Использование

### Создание нового приема

1. Выбрать пациента
2. Система автоматически определяет тип приема (первичный/повторный)
3. Выбрать шаблон (опционально)
4. Заполнить форму по секциям:
   - Тип приема и дата
   - Антропометрия
   - Анамнез
   - Показатели жизнедеятельности
   - Объективный осмотр по системам
   - Жалобы (для AI анализа)
5. AI автоматически анализирует данные и предлагает диагнозы
6. Открыть карточку заболевания из рекомендаций (в sidepanel)
7. Выбрать диагноз (ручной ввод/МКБ/база знаний)
8. Выбрать препараты из рекомендаций (с учетом аллергоанамнеза)
9. Завершить прием (показывается форма согласия при необходимости)

### Завершение приема

При завершении приема (`status: 'completed'`):
- Проверяется необходимость информированного согласия
- При необходимости показывается форма согласия
- После сохранения согласия прием завершается

## Миграция данных

Все новые поля в модели `Visit` опциональны для обратной совместимости. Существующие приемы будут работать без изменений.

Для миграции старых данных:
- `primaryDiagnosisId` → `primaryDiagnosis` (JSON с кодом МКБ и названием)
- `complicationIds` → `complications` (JSON массив)
- `comorbidityIds` → `comorbidities` (JSON массив)

## Безопасность

- Двойная валидация (frontend + backend)
- Обеспечение аутентификации во всех IPC handlers
- Аудит-логирование критичных действий
- Шифрование ПДн (существующая система)
- Транзакции для связанных операций

## Производительность

- Кеширование на нескольких уровнях
- Debounce для AI анализа (1.5 сек)
- Автоматическая отмена предыдущих AI запросов
- Оптимизация парсинга JSON полей (один раз на границе данных)
