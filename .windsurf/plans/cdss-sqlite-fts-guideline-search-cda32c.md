# SQLite FTS5: быстрый поиск по PDF chunks + merge с симптомами

Сделать офлайн-быстрый поиск диагнозов по жалобам/осмотру за счёт индекса SQLite FTS5 по PDF-chunks клинических рекомендаций и объединения результатов с текущим symptom-based поиском.

## Почему FTS5

- Работает полностью офлайн
- Очень быстрый поиск по большим текстам
- Ранжирование через `bm25()` «из коробки»
- Не требует AI/эмбеддингов

---

## 1) Хранение чанков: отдельная таблица `GuidelineChunk` (решение)

### Выбранный вариант: новая таблица `GuidelineChunk`

Создаём отдельную таблицу (через Prisma model или raw SQL):
- `GuidelineChunk`
  - `id` (PK)
  - `guidelineId` (FK → `ClinicalGuideline.id`)
  - `diseaseId` (FK → `Disease.id`) — денормализация для аггрегации
  - `pageStart`, `pageEnd` (int, optional)
  - `type` (`complaints|physical_exam|clinical_picture|diagnostics|treatment|other`)
  - `text`
  - `createdAt`

Плюсы:
- проще индексировать и обновлять
- не нужно постоянно парсить/пересериализовывать JSON
- легко вернуть «evidence chunks» в UI

---

## 1.1) pageStart/pageEnd: хранить или нет (рекомендация)

Рекомендация: **да, хранить `pageStart/pageEnd`**, но в «дешёвом» варианте.

- Зачем:
  - UI сможет показать врачу источник (страница) и дать быстрый переход к месту в PDF
  - проще отлаживать качество поиска (видно, где именно совпало)
  - полезно для AI-ранжирования (можно прикреплять более точный контекст)
- Как хранить оптимально:
  - Для простого chunking (по страницам/по fixed-size внутри страницы) достаточно одного поля `page`.
  - Для chunking, который может пересекать границы страниц (склеенный текст + fixed-size), лучше хранить:
    - `pageStart` и `pageEnd` (int)
- Если не хочется усложнять сейчас:
  - начать с `page` (или `pageStart`= `pageEnd`),
  - позже расширить схему до диапазона страниц без ломки API (поля уже будут).

### Вариант B (быстрый старт): оставить `ClinicalGuideline.chunks` (JSON) и строить FTS из него

Минусы:
- тяжёлые операции с JSON при обновлении/удалении
- труднее поддерживать целостность

## 2) Индекс: виртуальная таблица FTS5

Создаём FTS5 таблицу `GuidelineChunkFTS`:

- `text`
- `chunkId` (UNINDEXED)
- `diseaseId` (UNINDEXED)
- `guidelineId` (UNINDEXED)
- `type` (UNINDEXED)

Подходы:
- Стартовый и самый простой: **ручная синхронизация** (явные `INSERT/DELETE` в FTS)
- Более продвинутый: `content=GuidelineChunk` + triggers (можно позже)

---

## 3) Chunking «без смысловой нагрузки», но стабильный

Алгоритм:
- извлечь текст постранично
- нормализовать пробелы
- резать фиксировано:
  - `chunkSizeChars = 1000` (или 800–1200)
  - `overlapChars = 150`
- НЕ делать последующее `[:1000]` — длина контролируется нарезкой

Опционально дёшево: грубая типизация `type` по заголовкам/ключевым словам:
- встречается `"Жалобы"` → `complaints`
- `"Физикальное обследование"` → `physical_exam`
- иначе `other`

---

## 4) Построение FTS-запроса из визита (без AI)

1) Берём из визита:
- жалобы
- объективный осмотр (включая по системам)
- vitals (температура/SpO2/ЧДД/пульс)

2) Нормализуем:
- lower-case
- убрать пунктуацию
- убрать стоп-слова

3) Расширяем словарём (у тебя уже есть словарная инфраструктура):
- `мелкоточечная сыпь` → `экзантема`, `скарлатиноподобная`
- `гиперемия зева` → `фарингит`, `тонзиллит`
- `38.8` → `лихорадка`, `фебрильная`

4) Собираем FTS query:
- ключевые слова + несколько фраз

---

## 5) Поиск по PDF chunks

FTS запрос (идея):
- получить top-N чанков (например 50):
  - `SELECT chunkId, diseaseId, bm25(GuidelineChunkFTS) as score FROM GuidelineChunkFTS WHERE GuidelineChunkFTS MATCH ? ORDER BY score LIMIT 50`
- агрегировать по `diseaseId`:
  - `chunkScore(disease) = max(score)` или `sum(top3)`

Дополнительно:
- хранить top-2 чанка per disease как evidence

---

## 6) Merge с текущим symptom-based поиском

Делаем два пути параллельно:

- Path A: текущий `searchBySymptoms()` → `symptomScore`
- Path B: FTS chunks → `chunkScore`

Дальше:
- нормализовать score в [0..1]
- `finalScore = 0.6*chunkScore + 0.4*symptomScore`
- выдавать топ-10 diseases + evidence chunks

Fallback:
- если нет chunk индекса/чанков → только Path A
- если нет embeddings по симптомам → только Path B

---

## 6.1) AI как опциональный усилитель (не обязательный слой)

Базовый слой системы всегда работает на **FTS5 + keyword/словарь** (и при желании на текущем `searchBySymptoms()`), а AI подключается только если доступен интернет и валидный ключ.

### Что именно даёт AI

- Embeddings:
  - (опционально) эмбеддинг запроса визита (жалобы+осмотр) для семантического re-rank поверх FTS
  - (опционально) эмбеддинги disease-level полей (уже есть `symptomsEmbedding`)
- Ranking:
  - (опционально) `Gemini rankDiagnoses` получает короткий контекст:
    - клинические данные визита
    - топ диагнозов из merge (например top-8)
    - top-1/2 «evidence chunks» из FTS для каждого диагноза
  - возвращает `{diseaseId, confidence, reasoning}`

### Важное правило отказоустойчивости

- AI слой должен быть обёрнут в **circuit breaker** (у тебя он уже есть в нормализаторе симптомов):
  - если X ошибок подряд / таймауты / 401/403 — breaker открывается на N минут
- При любой ошибке AI:
  - логируем деградацию (например `logDegradation('rank', 'FTS')`)
  - возвращаем результаты базового слоя (FTS+merge) без падения пайплайна

### Схема деградации

1. `FTS + symptom search + AI rank` (если доступно)
2. `FTS + symptom search` (если AI rank недоступен)
3. `FTS only` (если symptom embeddings/AI недоступны)
4. `keyword fallback` (если FTS ещё не построен/нет данных)

## 7) Обновление индекса

Нужны операции:
- при загрузке/обновлении guideline:
  - удалить старые chunks/FTS строки
  - создать новые
- IPC `reindex`:
  - пересобрать всё (если поменяли алгоритм нарезки)
